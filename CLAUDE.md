> 이 문서는 Claude Code가 이 플랫폼을 개발/유지보수할 때 참고하는 프로젝트 명세입니다.
> 실험 설계·요구사항이 바뀌면 이 문서는 반드시 함께 갱신할 것.

## 1. 프로젝트 배경 (연구 개요)

**연구 주제**: 지하주차장 차량 경보음 기반 위치추정 오디오 데이터셋 구축

**문제 의식**
- 지하주차장은 GPS가 잡히지 않아 위치 기반 차량 안내가 불가능
- 사람들은 리모컨 경보음("삐빅")으로 차를 찾지만, 콘크리트 벽·기둥·차량 등의 심한
  다중 반사음 때문에 실제 차량 방향을 직관적으로 파악하기 어려움
- → 소리만으로 차량의 상대적 위치(전/후/좌/우, 가까움/멀음)를 추정하는 AI 모델 가능성 탐색

**연구 선정 이유**
1. 실생활 밀착형 문제 (필요성을 직관적으로 설명 가능)
2. 방법론이 명확하고 반복 실험 가능한 실험 → 학교 모집·별도 승인 없이 직접 비용 가능
3. 기존 연구 공백: 차량 경보음 × 지하주차장의 강한 다중 반사 조합을 다룬 사례가 드묾
   (관련 관련 논문 33편의 한계점을 15개 유형으로 분류함. 상위 한계 유형:
   상황·조건 다양성 부족 7편, 지역적 편향 5편, 실제 환경과의 괴리 5편,
   거리 조건 미세분화·통제 부족 4편, 표본 규모 부족 4편 등)

**연구 흐름**: 데이터셋 구축 → AI 모델 개발(전처리·특징추출·baseline) → 응용 검토
(차량 찾기 앱, 스마트 주차 시스템, 반사환경 위치추정 연구)

**발표자료에 명시된 실험 한계점**
- 차량 1~2대, 특정 주차장에서만 우선 구축 → 추후 다양한 차량·건물로 확장 필요
- 방향(전/후/좌/우) 세분화된 각도별 확장 필요
- 주차 위치·주변 장애물 등 환경 변수 통제가 어려움

## 2. 데이터 수집 계획

**일정**: 8/18 ~ 8/21, 하루 약 5시간

**장소 (동일)**
- 백화점: 현대아울렛 (평일 주차요금 무료 10:30~21:00)
- 마트: 홈플러스(주차료 확인 필요), 무지개 마트, 코스트코 마트(미정)

**차량**: 렌트. 세단 + 소나타(가솔린 연료). 렌트 상황에 따라 확정.

**실험 조건 (교수님 피드백 반영 후 확정)**

| 팩터 | 수준 | 개수 |
|---|---|---|
| 방향 | 전 / 후 / 좌 / 우 | 4 |
| 거리 | 근거리 / 중거리 / 원거리 | 3 |
| 차종 | 세단 / 기타 | 2 |
| 장소 | 백화점 / 마트 | 2 |
| 주차위치 | 전방주차 / 후방주차 | 2 |

- 총 조건: 4×3×2×2×2 = **96개**
- 조건당 반복: **30회** (중심극한정리, X̄ ~ N(μ, σ²/n), n ≥ 30)
- 총 목표 클립: **2,880개**
- 클립 길이: **6초 기본 (5~6초 허용)**
  - 초안 4초 계획. 근거: SVM 기준 6초/4초 성능차 통계적으로 비유의, 4초 미만은 유의한 성능 하락
  - 교수님 피드백으로 5~6초로 상향
- 수집 방식: 조건당 10분 이내, 마이크 녹음 앱 일정 간격으로 경보음 발생 → 파이썬(pydub/librosa) 커팅
- 규모 참고: ESC-50(~2.8h), STARSS23(~7.5h), TAU SSE 2019(~8.3h), UrbanSound8K(~9.7h)와 비교 설계

**교수님 피드백 (반드시 유지할 설계 결정)**
1. 클립 길이 4초 → 5~6초 상향 ✅
2. 기능없음/벽면없음을 나타내는 실험 "조건"에서 제외하는 "라벨"로는 반드시 기록 ✅
   (조건 384→96개 축소, 라벨은 추후 부산물으로 유지. 코드에서 이 구분을 절대 깨지 말 것)
3. 전방/후방 주차는 조건으로 유지 ✅
4. 데이터 저장·관리 플랫폼 개발 → **이 프로젝트가 바로 그 플랫폼**

## 3. 라벨링 체계

**조건 라벨 (커버리지 계산 대상)**: 방향, 거리, 차종, 장소, 주차위치

**추가 라벨 (조건화 제외, 클립마다 기록)**
| 항목 | 내용 |
|---|---|
| 층 | 지하 몇 층 (B1/B2/B3/기타) |
| 구역 | 주차장 내 구역 (예: A3, 자유 텍스트) |
| 기능없음 | 비상 직접 인근 기능 유무 (있음/없음) |
| 벽면없음 | 비상 직접 인근 벽면 유무 (있음/없음) |
| 배경잡음 | 비상 주변 잡음 (자유 텍스트) |

**메타데이터**: clip_id, 업로드 시각, 업로드 이름, 파일명, 클립 길이(초), 파일 경로

**파일명 규칙 (자동)**
```
장소_차종_방향_거리_기능없음_벽면없음_주차위치_NNN.wav
예: 백화점_세단_전_근거리_있음_있음_전방주차_001.wav
```
- 조건 + 기능/벽면 라벨은 언더바로 연결, 마지막 3자리는 동일 조합 내 순번
- 업로드 시 순번은 자동 생성 (사용자가 올린 원본 파일명은 무시하고 리네임)

## 4. 플랫폼 요구사항 (이번에 만들 것)

**한 줄 요약**: 팀원 전체가 웹으로 접속해서 → 수집 현황을 보고 ⇒비어있는 파일을 업로드·저장하고
→업로드된 오디오의 음향 시각화를 자동으로 확인할 수 있는 웹사이트.

### 4-1. 핵심 기능

**A. 수집 현황 대시보드**
- 96개 조건 각각의 수집 개수 / 목표(30개) 대비 진행률
- 전체 진행률 (총 2,880개 대비)
- 미채료 조건 목록 (부족한 순 정렬)
- 조건 2축 히트맵 (예: 방향×거리)으로 부족한 조합 시각화

**B. 오디오 업로드 + 라벨링**
- 웹 폼에서 조건 5개 + 추가 라벨 5개 선택/입력 후 파일 업로드
- 서버가 파일명 규칙에 따라 자동 리네임 후 저장
- 메타데이터는 DB에 기록
- (2차 목표) 여러 파일 일괄 업로드 + CSV 라벨 매칭
- (2차 목표) 긴 원본 비상 업로드 시 서버에서 6초 단위 자동 커팅

**C. 오디오 음향 시각화 (자동 생성)** — 이번 버전의 핵심 차별점
업로드된 각 클립에 대해 아래 시각화를 자동 생성·표시:
1. **파형 (waveform)** — 시간 영역 신호
2. **멜 스펙트로그램 (mel spectrogram)** — librosa.feature.melspectrogram + power_to_db
3. **피치 궤적 (F0 contour)** — librosa.pyin 사용 (경보음의 기본 주파수 추적)
4. **떨림/변동성 지표** — 다음 중 구현 가능한 것부터:
   - F0의 시간적 변동 (F0 표준편차, 프레임 간 변화율) — "떨림"의 1차 지표
   - RMS 에너지 곡선 (진폭 변동)
   - (선택) jitter/shimmer — parselmouth(Praat) 라이브러리 사용 시
- 시각화는 업로드 시 1회 생성해 PNG로 캐시 (매 조회마다 재계산하지 않기)
- 클립 상세 페이지: 오디오 플레이어(미리듣기) + 위 시각화들 + 라벨 정보

**D. 자료실 (팀 자료 저장)**
- 발표자료(PPT/PDF), 회의록, 참고 논문 등 문서 업로드/다운로드
- 간단한 목록 + 업로드 + 업로드 날짜 표시

**E. 전체 데이터 조회**
- 라벨별 필터링 (예: 장소=백화점, 방향=전 만 보기)
- CSV로 메타데이터 내보내기

### 4-2. 기술 스택 (2026-07-14 재확정: Drive/Sheets 백엔드)

- **Streamlit** 단일 앱 (팀이 Python에 익숙, 빠른 개발, 오디오/시각화에 궁합 좋음)
  - 페이지 구성: 대시보드 / 업로드 / 클립 탐색(시각화) / 자료실 / 전체 데이터
- **메타데이터 저장**: Google Sheets (SQLite 대신 채택). `clips`, `documents` 두 시트를
  DB 테이블처럼 사용 (core/db.py, gspread 기반). 서비스 계정으로 접근.
- **파일 저장**: Google Drive (서버 로컬 디스크 대신). `audio/`, `viz/`, `docs/` 폴더 3개를
  Drive에 만들고 서비스 계정에 편집자 권한 공유 (core/storage.py, google-api-python-client 기반).
  파일은 비공개로 유지하고, 재생/표시 시 서비스 계정 권한으로 서버가 직접 내려받아 전달.
- **오디오 분석**: librosa, soundfile, (선택) parselmouth
- **시각화**: matplotlib → PNG bytes 생성 후 Drive 업로드 (로컬 파일 캐시 없음)
- **인증**: core/gcp_auth.py에서 서비스 계정 JSON(Streamlit secrets)으로 Drive/Sheets 클라이언트 발급

> 이 구조로 확정한 이유: 목표가 "내 컴퓨터를 끄고 있어도 팀원 누구나 접속 가능한 공개 URL"이기
> 때문. Streamlit Community Cloud는 재시작 시 로컬 파일이 초기화되므로, 데이터를 Drive/Sheets에
> 두어야 로컬 컴퓨터를 상시 켜둘 필요 없이 배포할 수 있다.
> (중간에 한 번 "로컬 저장 + 사용자 지정 경로"로 롤백했었으나, 그 방식은 앱을 실행 중인
> 컴퓨터가 항상 켜져 있어야 한다는 제약이 있어 목표와 맞지 않아 다시 Drive/Sheets로 되돌림.)
> 저장 백엔드를 다시 바꾸더라도 core/storage.py, core/db.py 두 모듈만 교체하면 되도록 유지할 것.

### 4-3. 배포/접근

- 팀 전체가 URL로 접속 (회원가입 없이, 간단한 공용 비밀번호 게이트 정도만)
- **배포: Streamlit Community Cloud** (공개 URL, `https://<app-name>.streamlit.app`)
  - 오디오/시각화/메타데이터가 모두 Google Drive·Sheets에 있으므로 컨테이너 재시작이나
    로컬 컴퓨터를 꺼도 데이터 유실 없이 서비스가 유지됨.
  - 서비스 계정 키 등 민감 정보는 Streamlit Cloud의 Secrets 설정에만 저장 (git에는 절대 X).

## 5. 개발 우선순위

1. **MVP**: 업로드(파일명 자동 생성+DB 기록) + 대시보드(96조건 진행률) → 8/18 수집 시작 전 필수
2. **시각화**: 클립 탐색 페이지 (파형/멜스펙트로그램/F0/RMS + 미리듣기)
3. 자료실, 전체 데이터 필터/CSV 내보내기
4. 일괄 업로드 + 자동 커팅 통합
5. 히트맵, jitter/shimmer 등 고급 지표

## 6. 주의사항

- 조건(CONDITIONS)과 라벨(EXTRA_LABELS)의 구분은 교수님 피드백에 따른 설계 결정 →
  임의로 합치거나 옮기지 말 것 (config.py에 주석으로 명시)
- 실험 수치(96조건/30반복/6초) 변경 시 반드시 교수님 확인 후 이 문서와 config.py 동시 갱신
- 시각화 생성은 업로드 시점에 수행하고 캐시할 것 (2,880개 클립을 매번 실시간 계산하면 느림)
- 한글 파일명/라벨을 다루므로 모든 파일 IO에 UTF-8 명시 (encoding="utf-8")
- 서비스 계정 키(JSON)와 .streamlit/secrets.toml은 절대 git에 커밋하지 말 것 (.gitignore 처리됨)
- Google Sheets/Drive 폴더 공유 설정이 안 되어 있으면 앱이 core/ui_common.py의
  안내 메시지를 보여주고 멈춤 (SETUP.md 참고) — 신규 팀원 온보딩 시 이 문서부터 안내
